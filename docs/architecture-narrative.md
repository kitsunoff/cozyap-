# Cozy User Apps: Архитектурное повествование

Этот документ описывает архитектуру платформы Cozy User Apps — системы управления приложениями для конечных пользователей поверх Cozystack.

## Проблема

Cozystack использует модель «один CRD на тип приложения». Хотите Minecraft-сервер — вот вам CRD для Minecraft. Нужен PostgreSQL — другой CRD. Это создаёт проблемы:

- CRD — ресурсы уровня кластера. Они захламляют API-сервер.
- Для создания CRD нужны права cluster-admin. Обычному арендатору их не дашь.
- Каждый новый тип приложения требует нового CRD, контроллера, поддержки.

Нужна универсальная абстракция: один механизм для любых приложений.

## Решение: Шаблоны и Экземпляры

Платформа вводит два ключевых ресурса:

**ApplicationTemplate** — шаблон приложения. Описывает, как установить, обновить и удалить приложение. Это cluster-scoped ресурс, доступный всем арендаторам только для чтения. Создаётся оператором платформы или берётся из публичного каталога.

**Application** — экземпляр приложения. Конкретный Minecraft-сервер, конкретная база данных. Живёт в namespace арендатора. Арендатор создаёт Application, ссылаясь на ApplicationTemplate, и указывает свои параметры.

Пример: арендатор хочет игровой сервер. Он находит в каталоге шаблон `minecraft`, создаёт Application с именем `my-minecraft`, указывает `serverName: "ACME Gaming"`, `maxPlayers: 50`. Платформа берёт шаблон, подставляет параметры, выполняет хуки установки.

### Параметры

Шаблон определяет, какие параметры принимает приложение. Типы простые: строка, число, булев, перечисление, список, ссылка на Secret/ConfigMap. Никакой вложенности — плоская структура для простоты валидации и рендеринга в UI.

Каждый параметр несёт подсказки для интерфейса: метка, описание, плейсхолдер, порядок отображения, ограничения (min/max для чисел, варианты для перечислений).

### Хуки жизненного цикла

Шаблон описывает три хука:

- **install** — выполняется при создании Application
- **upgrade** — выполняется при изменении spec
- **remove** — выполняется при удалении Application

Каждый хук — это последовательность шагов. Шаг — контейнер с образом и командой. Платформа превращает эти шаги в Argo Workflow и выполняет в целевом кластере.

### Действия (Actions)

Помимо системных хуков, шаблон может определять произвольные действия: бэкап, рестарт, миграция. Пользователь создаёт ресурс Action, платформа выполняет соответствующие шаги. Это расширяемый механизм операций без хардкода в платформе.

## Среды выполнения (Environment)

Приложениям нужен кластер для работы. Вместо того чтобы заставлять пользователя разбираться в Kubernetes, платформа предоставляет абстракцию **Environment**.

Environment — это декларативное описание среды выполнения. Пользователь говорит: «Хочу production-среду с узлами размера medium». Платформа:

1. Создаёт managed Kubernetes через Cozystack API
2. Устанавливает необходимые контроллеры (Argo Workflows, Argo Events, Ingress, cert-manager)
3. Генерирует kubeconfig и кладёт в Secret
4. Мониторит здоровье каждые 30 секунд

Application ссылается на Environment по имени: `targetEnvironment: production`. Контроллер находит Environment, берёт kubeconfig, создаёт Workflow в целевом кластере.

### Мультитенантность

Cozy User Apps не изобретает свою изоляцию. Она переиспользует существующую модель Cozystack:

- Каждый арендатор работает в своём namespace
- ServiceAccount с RBAC ограничивает доступ
- ResourceQuota лимитирует ресурсы
- NetworkPolicy изолирует трафик между арендаторами

В рамках своего namespace арендатор может создавать Environment, Application, Action. Между namespace'ами доступа нет. ApplicationTemplate — исключение: они cluster-scoped и read-only для всех.

### Размеры узлов

Environment поддерживает подсказки по размеру:

| Размер | vCPU | RAM |
|--------|------|-----|
| small | 2 | 4Gi |
| medium | 4 | 8Gi |
| large | 8 | 16Gi |
| xlarge | 16 | 32Gi |

Если не указать — платформа решит сама на основе нагрузки.

## Архитектура контроллеров

Единый бинарник `cozy-apps-operator` содержит несколько контроллеров:

**Environment Controller** — следит за Environment, создаёт кластеры через Cozystack, устанавливает контроллеры, мониторит здоровье.

**Application Controller** — следит за Application, валидирует параметры, создаёт Workflow для install/upgrade/remove, отслеживает их статус.

**Action Controller** — следит за Action, валидирует, создаёт Workflow для действий.

**Cleanup Controller** — удаляет завершённые Action через 24 часа.

### Машины состояний

**Environment**:
```
Pending → Provisioning → Ready ⇄ Degraded → Deleting → [удалён]
```

**Application**:
```
Pending → Installing → Running → Upgrading → Running
                   ↓              ↓
                Failed ←──────────┘
```

**Action**:
```
Pending → Running → Succeeded/Failed → [удалён через 24ч]
```

### Многокластерная архитектура

Контроллер работает в host-кластере (Cozystack), но создаёт ресурсы в Environment-кластерах. Для этого поддерживается кэш клиентов:

- Клиенты создаются лениво при первом обращении
- Kubeconfig обновляется каждые 5 минут
- Устаревшие клиенты (Environment удалён) собираются garbage collector'ом

### Финализаторы

Environment и Application имеют финализаторы для корректной очистки:

- Environment финализатор ждёт удаления всех Application, потом удаляет кластер
- Application финализатор запускает remove hook перед удалением

Action финализатора не имеет — это короткоживущий ресурс.

## DAG Engine: Argo Workflows

Для выполнения хуков и действий платформа использует Argo Workflows. Это зрелый DAG-движок с поддержкой:

- Параллельного и последовательного выполнения шагов
- Зависимостей между шагами (dependsOn)
- Retry с backoff
- Таймаутов на уровне шага и workflow
- Shared volume для передачи данных между шагами

### Маппинг шагов в Workflow

Каждый шаг из ApplicationTemplate превращается в DAG task:

```yaml
# В шаблоне
steps:
  - name: deploy
    image: alpine/helm:3.14
    args: [helm, install, ...]
    dependsOn: [prepare]

# Становится Argo Workflow DAG task
tasks:
  - name: deploy
    template: deploy
    dependencies: [prepare]
```

### Платформенные данные

Каждый Workflow получает доступ к:

- `/platform/kubeconfig` — kubeconfig для доступа к host-кластеру (для заказа managed-сервисов)
- `/platform/values.yaml` — параметры Application
- `/platform/app.yaml` — метаданные Application

Это позволяет шагам взаимодействовать с Cozystack для заказа баз данных, S3-бакетов и прочего.

### Модель выполнения

**Системные хуки** (install, upgrade, remove) — строго последовательные. Один Workflow на Application в один момент времени. Контроллер ждёт завершения перед запуском следующего.

**Действия** (Actions) — параллельные. Несколько Action могут выполняться одновременно. Ответственность за конфликты — на пользователе.

### Очистка

Workflow сохраняются для отладки:
- Успешные — 1 час
- Неудачные — 24 часа

Argo Workflows garbage collector удаляет их автоматически.

## Пользовательские сценарии

Платформа обслуживает разные категории пользователей:

### Конечный пользователь (End User)

Клиент ISP, хочет запустить игровой сервер. Kubernetes не знает, знать не хочет.

1. Открывает веб-интерфейс, логинится
2. Просматривает каталог: игры, базы данных, веб-приложения
3. Выбирает Minecraft, видит описание и параметры
4. Выбирает Environment (production или staging)
5. Заполняет форму: имя сервера, максимум игроков, режим игры
6. Нажимает «Deploy», смотрит на прогресс установки
7. Получает работающий сервер

Для отладки:
- Централизованные логи в Grafana
- Скачивание kubeconfig для kubectl exec/port-forward
- Просмотр логов Workflow при ошибках установки

### Автор шаблонов (Template Author)

Контрибьютор сообщества, упаковывает приложения в шаблоны.

1. Изучает, как приложение обычно устанавливается (Helm? Манифесты?)
2. Выделяет параметры, которые должен настраивать пользователь
3. Пишет хуки install/upgrade/remove
4. Добавляет действия (backup, restart)
5. Тестирует локально
6. Отправляет PR в репозиторий каталога

Best practices:
- Использовать известные базовые образы (alpine/helm, bitnami/kubectl)
- Обеспечить идемпотентность (install должен работать при частичном применении)
- Очищать все ресурсы в remove hook
- Не использовать latest-теги — пинить версии
- Не хардкодить то, что должно быть параметром

### Оператор ISP

Управляет платформой Cozystack, онбордит арендаторов.

1. Разворачивает Cozystack
2. Устанавливает cozy-apps-operator
3. Настраивает синхронизацию каталога шаблонов
4. Создаёт namespace для арендаторов, настраивает RBAC и квоты
5. Мониторит через алерты: Environment Degraded, Application Failed, Resource Exhaustion

### Разработчик

Использует платформу как среду разработки.

Git-based workflow:
1. Указывает Git-репозиторий как источник
2. Платформа клонирует, билдит через Kaniko, деплоит
3. Push в Git триггерит upgrade
4. Preview-окружения для pull request'ов

Database provisioning:
1. Шаблон создаёт Postgres CR в host-кластере через platform kubeconfig
2. Ждёт готовности
3. Копирует credentials в Environment
4. Деплоит приложение с секретом

## Каталог шаблонов

ApplicationTemplate нужно где-то хранить. Два варианта:

**Вариант A: etcd (стандартный CRD)**

Простота: это обычный Kubernetes CRD. Минус: etcd имеет лимит ~1.5MB на объект, нет версионирования, большой каталог нагружает etcd.

**Вариант B: Aggregated API Server + PostgreSQL** (рекомендуется)

Kubernetes Aggregation Layer позволяет подменить обработку определённой API-группы своим сервером. ApplicationTemplate обрабатывается `cozy-apps-apiserver`, который хранит данные в PostgreSQL.

Преимущества:
- Нет лимитов etcd
- Нативное версионирование: каждая версия шаблона — отдельная запись
- SQL-запросы для поиска и фильтрации
- Масштабируется независимо от etcd
- PostgreSQL уже есть в Cozystack

При этом Application, Environment, Action остаются стандартными CRD в etcd — они небольшие и их не так много.

### Версионирование шаблонов

Каждый ApplicationTemplate имеет версии. В первой версии Application ссылается на шаблон по имени без версии — всегда используется последняя. В будущем: `templateRef: minecraft@v1.2.3`.

Устаревшие версии можно пометить deprecated — они скрываются в UI, но продолжают работать для существующих Application.

## Сеть

Приложениям нужен внешний доступ. Environment включает сетевые контроллеры:

**Ingress Controller** (nginx) — HTTP(S) роутинг, TLS termination. Приложения создают стандартные Ingress-ресурсы.

**cert-manager** — автоматические TLS-сертификаты от Let's Encrypt.

**Cloud Controller Manager** — для LoadBalancer-сервисов. Minecraft-сервер на 25565 порту? Service типа LoadBalancer получит внешний IP.

**External-DNS** (опционально) — автоматическое создание DNS-записей.

### Namespace для приложений

Каждое приложение живёт в своём namespace внутри Environment:

```
Environment Cluster
├── argo-workflows/      # Платформа
├── cert-manager/        # Платформа
├── ingress-nginx/       # Платформа
├── my-minecraft/        # Приложение
├── my-webapp/           # Приложение
└── my-database/         # Приложение
```

Network Policy между приложениями пока не применяется — они могут общаться свободно. Это упрощение для первой версии.

## Наблюдаемость

Логи и метрики собираются централизованно:

- В каждом Environment работает monitoring agent
- Данные отправляются в центральную Grafana/Loki
- Арендатор видит логи своих приложений через Grafana-дашборд

Для интерактивной отладки — kubeconfig Secret позволяет подключиться к Environment напрямую и использовать kubectl.

## Резюме архитектуры

```
┌─────────────────────────────────────────────────────────────────┐
│                        Host Cozystack                            │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │                   cozy-apps-operator                        ││
│  │  Environment Controller | Application Controller            ││
│  │  Action Controller      | Cleanup Controller                ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                  │
│  ┌─────────────────┐  ┌─────────────────┐                       │
│  │ tenant-acme     │  │ tenant-globex   │                       │
│  │ ┌─────────────┐ │  │ ┌─────────────┐ │                       │
│  │ │ Environment │ │  │ │ Environment │ │                       │
│  │ │ Application │ │  │ │ Application │ │                       │
│  │ │ Action      │ │  │ │ Action      │ │                       │
│  │ └─────────────┘ │  │ └─────────────┘ │                       │
│  └─────────────────┘  └─────────────────┘                       │
│                                                                  │
│  ApplicationTemplate (cluster-scoped, read-only)                │
└─────────────────────────────────────────────────────────────────┘
          │                           │
          ▼                           ▼
┌─────────────────┐         ┌─────────────────┐
│ Environment     │         │ Environment     │
│ Cluster (ACME)  │         │ Cluster (Globex)│
│ ┌─────────────┐ │         │ ┌─────────────┐ │
│ │Argo Workflow│ │         │ │Argo Workflow│ │
│ │Ingress      │ │         │ │Ingress      │ │
│ │cert-manager │ │         │ │cert-manager │ │
│ │Applications │ │         │ │Applications │ │
│ └─────────────┘ │         │ └─────────────┘ │
└─────────────────┘         └─────────────────┘
```

Ключевые принципы:

1. **Единообразие**: один механизм для любых приложений
2. **Изоляция**: арендаторы в своих namespace, приложения в своих кластерах
3. **Декларативность**: Environment и Application — желаемое состояние, контроллеры приводят к нему
4. **Расширяемость**: новые приложения — это просто новые шаблоны, не код
5. **Переиспользование**: Cozystack multi-tenancy, Argo Workflows, стандартные Kubernetes-примитивы
