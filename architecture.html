Cozy User Apps

Архитектурное повествование — реактивная reconcile-модель на основе атомов

Сдвиг архитектуры

Исходный дизайн (ADR-0001–0007) описывал Cozyap как «менеджер пакетов поверх Argo»: каждый ApplicationTemplate нёс четыре императивных хука — install, upgrade, remove и suggestUpgrade. Контроллер выбирал нужный хук в зависимости от состояния CR.

Три давления потребовали переработки ещё до того, как был написан первый код:

  • Drift heal не был первоклассной задачей. Периодическое повторное применение требовало пятого типа workflow.
  • Монолитные шаблоны блокировали замену бэкенда. Шаблон «WordPress» жёстко включал MySQL, контейнер, Ingress и TLS в один неделимый блок.
  • Семантика хуков расплывалась по ADR. Spec шагов переопределялся трижды; ServiceAccount cluster-admin давал авторам шаблонов неограниченную власть.
Ключевой инсайт: install / upgrade / remove / suggestUpgrade / drift heal — это одна и та же операция: привести фактическое к желаемому. Они схлопываются в единый reconcile.
Старая модель (ADR-0001–0007)Новая модель (ADR-0008+)
Хуки install / upgrade / removeЕдиный reconcile
Монолитный ApplicationTemplateГраф атомов + ApplicationTemplate
Argo Workflows как DAG-движокTemporal или Pods с stdin/stdout
suggestUpgrade как отдельный хукКаскад через fingerprint
Drift heal — отдельная задачаReconcile по умолчанию (cron 4–24ч)
cluster-admin SA для workflowМинимальный RBAC, scoped по atom

Атомы

Атом — это версионированный, пригодный для сообщества строительный блок. Всё, что больше одного Kubernetes-объекта, строится из атомов. Атомы — чёрные ящики: их внутренние k8s-объекты не видны тенантам.

Атом объявляет:

  • Типизированные входные порты — что ему нужно от соседних атомов
  • Типизированные выходные порты — что он публикует для потомков
  • Схему параметров — что настраивает пользователь
  • Reconcile-логику — единственная lifecycle-операция
  • Status-логику — read-only health probe, публикует named conditions

AtomTemplate

AtomTemplate — кластерный CRD. Хранит определение атома: образ executor'а, объявления портов, схему параметров, таймауты reconcile и status. Тенанты читают его, не пишут.

apiVersion: cozyapps.cozystack.io/v1alpha1
kind: AtomTemplate
metadata:
  name: postgres
  labels:
    cozyapps.cozystack.io/version: "1.2.0"
spec:
  displayName: "PostgreSQL"
  category: database

  inputs:
    backupBucket: { type: secret-ref, required: false }
  outputs:
    credentials: { type: secret-ref }
    service:     { type: service-ref }

  params:
    - name: version
      type: enum
      enum: ["14", "15", "16"]
      default: "16"
    - name: replicas
      type: int
      default: 1

  # Executor image. Invocation mechanism is in ADR-0010 (Temporal)
  # or ADR-0011 (Pods stdin/stdout).
  executor:
    image: ghcr.io/cozystack/cozyap-atoms/postgres:1.2.0
    resources:
      requests: { cpu: 100m, memory: 128Mi }
    requires:
      dataplaneAccess: true

  reconcile:
    timeout: 30m
    cron: 4h

  status:
    cron: 60s
    publishes: [ClusterReady, BackupOk, StorageHealthy]

  actions:
    - name: backup
      displayName: "Create backup"

AtomInstance

AtomInstance — namespace-scoped CRD, создаётся контроллером для каждой вершины в графе Application. Тенант не пишет его напрямую — только через родительский Application.spec.

apiVersion: cozyapps.cozystack.io/v1alpha1
kind: AtomInstance
metadata:
  name: my-app-postgres
  namespace: tenant-acme
  ownerReferences:
    - kind: Application
      name: my-app
spec:
  templateRef:
    name: postgres
    version: "1.2.0"        # явный patch-pin
  params:
    version: "16"
    replicas: 3
  inputs:
    backupBucket:
      from:
        atom: my-app-s3
        port: bucketSecret
status:
  phase: Ready
  conditions:
    - type: ClusterReady
      status: "True"
      state: ok
      reason: ClusterHealthy
      message: "all 3 instances streaming WAL"
  outputs:
    credentials:
      ref: { kind: Secret, name: my-app-postgres-credentials }
      fingerprint: "sha256:9b74c9897bac…"
      observedAt: 2026-05-20T10:32:00Z

Типизированные порты

13 типов формируют v1-словарь. Каждый порт — входной или выходной — объявляется с одним из них:

ТипНесётBacked by
secret-refСсылка на Secret (с опциональным ключом)Secret
tls-secret-refСсылка на TLS SecretSecret (tls)
configmap-refСсылка на ConfigMapConfigMap
service-refСсылка на Service (с опциональным портом)Service
pvc-refСсылка на PVCPersistentVolumeClaim
serviceaccount-refСсылка на ServiceAccountServiceAccount
workload-refСсылка на Deployment/StatefulSetDeployment/StatefulSet
image-refОбраз контейнера с тегом/digeststring (semantic)
ingress-hostHostname для Ingress-правилаstring (semantic)
stringПроизвольная UTF-8 строкаscalar
numberЧислоscalar
booleanБулевscalar
anyEscape hatch, opaque payloadbytes (base64)

Subtyping (однонаправленное):

tls-secret-ref  →  secret-ref
image-ref       →  string
ingress-host    →  string

Обратное не работает: secret-ref не удовлетворяет tls-secret-ref. Сиблинги не взаимозаменяемы. any — универсальный sink, но никогда не source для типизированного input.

Output materialisation

Внутренние k8s-объекты атома — детали реализации. Контрактная поверхность — AtomInstance.status.outputs. Все ссылки — в том же namespace. Атом владеет объектами, к которым ссылается; контроллер устанавливает ownerRef от k8s-объекта к AtomInstance.

Каждый output содержит:

  • ref — kind, name, и опциональный key/port
  • fingerprint — SHA-256 от материализованного содержимого (не resourceVersion)
  • observedAt — timestamp последней записи

Каскадная детекция

fingerprint — примитив детекции изменений. Это стабильный хэш материализованного содержимого — не resourceVersion объекта (он растёт при нерелевантных ресинках).

Когда AtomInstance.status.outputs.<port>.fingerprint меняется, контроллер enqueue'ит reconcile всех downstream AtomInstance, чьи spec.inputs.<port>.from ссылаются на этот output. Одинаковые fingerprint (no-op) не распространяются. Это каскад — чистый controller watch loop, без Argo Events.

Граф приложений

Application — namespace-scoped CR в тенантском namespace. Описывает граф вызовов атомов и параметры от пользователя. Контроллер создаёт по одному AtomInstance для каждой вершины графа.

Composition

ApplicationTemplate — версионированный граф атомов плюс форма от UserInput-атома. Автор шаблона соединяет атомы в builder UI или YAML; результат портируем между кластерами.

ApplicationTemplate "wordpress"
├── UserInput          → params.hostname, params.adminPassword
├── Postgres (1.2.0)   ← backupBucket ← S3 outputs.bucketSecret
│   outputs: credentials (secret-ref), service (service-ref)
├── S3 (1.0.4)
│   outputs: bucketSecret (secret-ref)
├── Container (2.1.0)  ← db ← Postgres.credentials
│                      ← dbHost ← Postgres.service
│   outputs: image (image-ref), workload (workload-ref)
├── Ingress (1.3.0)    ← backend ← Container.workload
│                      ← tls ← TLSCert.cert
│   outputs: host (ingress-host)
└── TLSCert (1.1.0)    ← hostname ← UserInput.hostname
    outputs: cert (tls-secret-ref)

UserInput и Constant

UserInput — специальный системный атом, не имеющий executor'а. Его выходные порты — значения из Application.spec.values. Он формирует форму для пользователя в UI.

Constant — атом с захардкоженным значением в шаблоне. Использутся когда автор шаблона не хочет давать пользователю возможность настраивать конкретный параметр.

Reconcile

Reconcile — единственная lifecycle-операция. Заменяет install, upgrade, remove, suggestUpgrade и drift heal. Атом получает на вход текущее желаемое состояние и приводит к нему фактическое.

Триггеры

  • Cron — safety net каждые 4–24 часа (настраивается в AtomTemplate.spec.reconcile.cron)
  • Изменение spec — пользователь изменил Application.spec.values
  • Upstream cascade — fingerprint output'а upstream-атома изменился

Удаление

Удаление обнаруживается по deletionTimestamp внутри reconcile-логики. Финализатор удерживает CR до тех пор, пока reconcile не завершится чисто. Нет отдельного хука remove — только один reconcile, который проверяет deletion и убирает всё созданное.

Execution Layer

Выбор execution layer — открытый вопрос (ADR-0012). Два варианта остаются на рассмотрении; контракт атома (ADR-0009) одинаков для обоих.

Option A: Temporal (ADR-0010)

Атомы — это Temporal workflows + activities, упакованные в worker-образы. Один shared Temporal-кластер на host-стороне; atom workers работают в tenant control plane; reconcile и status — типы workflow, запускаемые оператором.

Топология
  • Shared Temporal cluster (host) — один на всех тенантов
  • Temporal namespace per tenant — изоляция истории workflow
  • Atom worker Deployment per (tenant, AtomTemplate) — регистрирует workflow types
  • Scale-to-zero через KEDA ScaledObject на глубину task queue
  • Platform worker (cozy-system) — для host-side операций

Плюсы: нативная durability (workflow переживает перезапуск worker'а), transition tracking через Temporal activity IDs, Temporal UI с полной историей.

Минусы: Temporal cluster — дополнительный stateful-компонент, широкий blast radius (outage затрагивает всех тенантов), крутая learning curve для авторов атомов.

Option B: Pods with stdin/stdout (ADR-0011)

Атомы — контейнерные образы с entrypoint cozyap-atom reconcile. Оператор запускает Job на каждый reconcile trigger. Нет persistent worker-процессов. Durability DIY через transitionId в AtomInstance.status.

JSON I/O контракт
# stdin payload (оператор → атом)
{
  "run": { "type": "reconcile" },
  "params": { "version": "16", "replicas": 3 },
  "inputs": { "backupBucket": { "kind": "Secret", "name": "…" } },
  "environment": { "dataplaneKubeconfig": "…" },
  "state": { "transitionId": "migrate-pg14-to-pg16-done" }
}

# stdout result (атом → оператор)
{
  "outputs": { "credentials": { "kind": "Secret", "name": "…" } },
  "conditions": [ { "type": "ClusterReady", "status": "True", … } ],
  "state": { "transitionId": "migrate-pg14-to-pg16-done" }
}

Плюсы: нет внешнего сервера, нулевое потребление в idle, любой язык (достаточно stdlib), узкий blast radius (сломанный атом валит только свой Job).

Минусы: durability DIY — atom author обязан правильно реализовать transitionId, долгоживущие операции требуют re-entrant логики через несколько Job-вызовов.

Статус и условия

Status — отдельный, read-only workflow per atom. Работает чаще, чем reconcile (30–60 s cron), публикует список типизированных условий в Application.status.atoms[<name>].conditions.

Схема conditions

Каждый элемент AtomInstance.status.conditions следует k8s Condition API с одним расширением — полем state:

ПолеТипПримечание
typeCamelCase stringОбъявлено в AtomTemplate.spec.status.publishes + platform-reserved
status"True"/"False"/"Unknown"k8s-стандарт, kubectl wait --for=condition=… работает
stateok/warn/error/info/unknownРасширение для UI; более тонкая градация чем True/False
reasonCamelCase stringMachine-readable метка
messagestringОдно короткое предложение
lastTransitionTimeRFC3339Устанавливает контроллер при переходе
observedGenerationint64Generation spec при последнем обновлении

Reserved conditions (пишет платформа, не автор атома):

  • Reconciled — результат последнего reconcile run
  • InputsResolved — все spec.inputs привязаны к существующим upstream outputs
  • Deleting — deletionTimestamp установлен, удаление в процессе

Агрегация

Агрегированное состояние AtomInstance — наихудшее из условий по правилу: error > warn > info > ok > unknown. Агрегированное состояние Application — наихудшее среди всех его AtomInstance.

Условие обязательно только если атом объявил его в AtomTemplate.spec.status.publishes. Отсутствующее условие трактуется как unknown.

Каталог и версионирование

Каталог хранит два типа артефактов: AtomTemplate и ApplicationTemplate, оба версионированы semver.

  • AtomTemplate версионируется semver. Версия — в label (cozyapps.cozystack.io/version), не в имени объекта.
  • ApplicationTemplate пинит атомы по patch-версии. postgres@1.2.0 — явный pin. Нет ranges, нет implicit upgrades.
  • Breaking changes (удалён output, переименован порт, изменён тип параметра) → major bump.
  • Добавление outputs, conditions, optional параметров → minor bump.
  • Внутренние изменения (другой operator backend, bug fix в reconcile) → patch bump.

Каталог синхронизируется из git-репозитория (ADR-0016). Версии иммутабельны: опубликованный postgres@1.2.0 никогда не меняется. Каталожный UI показывает флаг «newer version available» на затронутых ApplicationTemplate.

Ключевые принципы

  • 1 Единственная операция: reconcile покрывает install, upgrade, remove, drift heal и внешние каскады
  • 2 Чёрный ящик: внутренние объекты атома не видны тенанту; контрактная поверхность — только типизированные outputs и conditions
  • 3 Замена бэкенда бесплатна: атом с service-ref работает поверх CNPG, Zalando или внешнего RDS — граф не меняется
  • 4 Каскад без шины: fingerprint diff в controller watch loop — без Argo Events, без Kafka
  • 5 Executor-neutral контракт: типы портов, output materialisation, conditions одинаковы для Temporal и Pods
