Problem

Two Problems, One Root Cause

Cozystack today

❌One CRD per app type — Postgres, Redis, Minecraft, WordPress…
❌CRDs are cluster-wide — every new app pollutes the API server
❌Adding a new app type requires cluster-admin
❌Tenants cannot create their own app types

Old Cozyap design (ADR-0001–0007)

❌install / upgrade / remove hooks — drift heal was not built in
❌Monolithic templates — replacing one DB backend required rewriting the template
❌Argo Workflows as DAG engine — runs once, archives; we need a continuous reconciler
❌cluster-admin SA per workflow — unscoped power for template authors
Root cause: imperative lifecycle hooks model. Each lifecycle event is a different operation — but they're all the same thing: make actual match desired.

Building Block

The Atom

Atom
versioned · community-writable · black box
Typed input ports — what it needs from siblings
Typed output ports — what it exposes downstream
Parameter schema — what users configure
reconcile — the only lifecycle operation
status — read-only health probe (30–60s cron)
Actions — user-triggered one-shots (backup, restart…)
Black box: internal k8s objects are not visible to tenants. The contract surface is typed outputs + named conditions only.
AtomTemplate
Cluster-scoped CRD. Read-only for tenants. Written by catalog. Semver-versioned.
AtomInstance
Namespace-scoped. Created by the controller per graph node. Holds outputs + conditions in status.
Application
A wired graph of atoms. Tenant-owned. References an ApplicationTemplate version.

Cascade

Fingerprint-Driven Cascade

Every output carries a fingerprint — SHA-256 of the materialised content, not resourceVersion.

1
Postgres reconcile finishes. Rotates the credentials Secret.
2
Writes new fingerprint to AtomInstance.status.outputs.credentials.
3
Controller diffs fingerprint — changed. Enqueues reconcile for every downstream AtomInstance that binds this output.
4
Container atom reconciles, picks up the new credential. No manual trigger needed.
No Argo Events. No Kafka. Pure controller watch loop on AtomInstance.status.
Fingerprint rules
secret-ref → SHA-256 of Secret data
service-ref → SHA-256 of {clusterIP, ports}
image-ref → SHA-256 of resolved digest
string/number → SHA-256 of value
Identical fingerprints do not propagate — no-op reconciles don't cascade.

AtomTemplate

AtomTemplate: Postgres

apiVersion: cozyapps.cozystack.io/v1alpha1 kind: AtomTemplate metadata: name: postgres labels: cozyapps.cozystack.io/version: "1.2.0" spec: inputs: backupBucket: { type: secret-ref, required: false } outputs: credentials: { type: secret-ref } service: { type: service-ref } params: - name: version type: enum enum: ["14", "15", "16"] default: "16" executor: image: ghcr.io/cozystack/cozyap-atoms/postgres:1.2.0 # invocation: ADR-0010 (Temporal) or ADR-0011 (Pods) reconcile: timeout: 30m cron: 4h status: cron: 60s publishes: [ClusterReady, BackupOk, StorageHealthy] actions: - name: backup displayName: "Create backup"
Executor-neutral
The executor block carries only image + resources. Invocation mechanism (Temporal workflowType vs stdin/stdout command) is added by ADR-0010 or 0011.
Versioning
Semver on a label. Same metadata.name across versions. Running apps continue pinned at 1.2.0 after 1.2.1 ships.
Status publishes
Named conditions are the atom's public API. Removing or renaming them is a major version bump.

Status API

Status & Conditions

Each AtomInstance carries conditions following the k8s Condition API with one extension — the state field.

status: phase: Ready conditions: - type: Reconciled # platform-reserved status: "True" state: ok reason: ReconcileSucceeded message: "no changes applied" - type: ClusterReady # atom-declared status: "True" state: ok reason: ClusterHealthy message: "3 instances streaming WAL" - type: BackupOk # atom-declared status: "False" state: warn reason: BackupStale message: "no backup in last 36h"
state field
okall good
warndegraded but running
errorbroken
infotransient
unknownnot probed yet
Aggregation: worst state wins
error > warn > info > ok > unknown
kubectl-compatible: standard status field means kubectl wait --for=condition=ClusterReady just works.

Execution Layer

Two Options, One Atom Contract

The atom contract (ports, outputs, conditions) is identical for both. Only the invocation mechanism differs.

Option A — Temporal (ADR-0010)
✓ Durable workflows survive worker restart
✓ Transition tracking via activity IDs — no re-execution of committed side effects
✓ Temporal UI: full workflow history per tenant
✓ Native long-running operation support
✗ Temporal cluster: stateful operational dependency
✗ Wide blast radius — outage suspends all tenants
✗ Atom authors must learn Temporal determinism rules
Option B — Pods stdin/stdout (ADR-0011)
✓ No external server — only Kubernetes Jobs
✓ Zero idle resource cost — Jobs are ephemeral
✓ Any language: read stdin, write stdout, exit 0
✓ Narrow blast radius — broken atom fails its own Job only
✗ Durability is DIY via transitionId in status
✗ Long-running ops need re-entrant logic across Job invocations
✗ stdout contamination = operator parse failure
Decision pending (ADR-0012). Both options remain Proposed. The decision is made before any atom SDK work begins.

Option B

Pods with stdin/stdout

1
Reconcile trigger fires. Operator creates a Kubernetes Job.
2
JSON payload (params, inputs, kubeconfig, previous state) written to stdin.
3
Atom runs reconcile logic. Writes JSON result to stdout. Logs to stderr.
4
Job exits 0. Operator reads stdout, patches AtomInstance.status.
Any language. The atom just needs to read stdin and write stdout. No SDK required beyond the JSON contract.
# stdout result { "outputs": { "credentials": { "kind": "Secret", "name": "my-app-postgres-creds" } }, "conditions": [{ "type": "ClusterReady", "status": "True", "state": "ok", "reason": "Healthy", "message": "3 instances streaming WAL" }], "state": { // durability token — passed back // on next invocation so atom // can skip committed side effects "transitionId": "migrate-pg14-to-pg16-done" } }

Atom Author Flow

Writing a New Atom

01
Declare typed input/output ports in AtomTemplate
02
Implement reconcile — idempotent, detects deletion via deletionTimestamp, returns typed ReconcileResult
03
Implement status — read-only health probe, publishes every declared condition on every run
04
Add actions for user-triggered one-shots: backup, restore, restart
05
Run catalog linter — idempotence check (reconcile twice, expect same fingerprints), condition completeness
06
Publish to catalog with semver tag — available in all ApplicationTemplates immediately
Must guarantee
Idempotence
Output stability
Condition completeness
No cross-namespace writes
Cleanup on deletion
Higher barrier than old "write three hooks" model — but enables backend swap and drift heal by design.

Summary

Reactive Application Platform

Five principles that make the design work.

01
One Operation
reconcile replaces install / upgrade / remove / drift heal. Triggered by cron, spec change, or upstream fingerprint diff.
02
Composable Atoms
Typed ports + fingerprint outputs. Backend swap is free — the graph doesn't change when CNPG becomes Zalando.
03
Cascade Without a Bus
Fingerprint diff in controller watch loop. No Argo Events, no Kafka. No-op reconciles don't propagate.
04
Executor-Neutral Contract
Atom ports, outputs, conditions are the same for Temporal and Pods. Execution layer decision doesn't change the atom API.
05
Drift Heal by Default
Cron reconcile runs every 4–24h on every atom. Drift is corrected automatically, not as a special feature.
